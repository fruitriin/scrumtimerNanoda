# タスク完了パターン — dev-loop の実践知見

## 発見した知見

### 1サイクルの時間見積もり

| タスク規模 | 例 | 目安時間 |
|---|---|---|
| 小（1ファイル修正） | Plan 0011 lint エラー修正 | 5-10分 |
| 中（テスト追加・リファクタ） | Plan 0008 テストカバレッジ、Plan 0013 バンドル最適化 | 15-25分 |
| 大（新機能追加） | Plan 0005 メモ機能 | 30-45分 |

Stage 2（レビュー）が最も時間を消費する（各エージェント 20-80秒）。

### タスク選択の優先順位

1. **ビルドを壊す問題** → 即修正（lint エラー、型エラー）
2. **複利効果** → テスト基盤、ツール改善、バンドル最適化
3. **若番** → 機能タスクは番号順

### 品質計画の自動生成パターン

TODO が空になったら、Explore エージェントでコードベースを調査して品質課題を洗い出し、
計画ファイルを作成する。調査観点:

- バンドルサイズ（`vp build` 出力）
- E2E テストカバレッジ
- アクセシビリティ（ARIA、タッチターゲット）
- エラーハンドリング
- レスポンシブ対応

### 完了処理のチェックリスト

1. Plan ファイルに「実装結果」セクション追記
2. TODO.md の状態を「完了」に更新 + アーカイブに追加
3. Progress.md をアーカイブ → テンプレートから新規作成
4. Feedback.md に問題・改善アクションがあれば追記
5. knowhow に知見があれば記録
6. コミット

### knowhow 記録の3タイミング

| タイミング | 記録内容 |
|---|---|
| **実装中** | コーディングパターン、ライブラリの癖、ハマりポイント |
| **レビュー後** | エージェントが検出した問題パターン、運用上の気づき |
| **タスク総括時** | タスク選択判断、時間配分、プロセス改善 |

## プロジェクトへの適用

- dev-loop スキル (`/addf-dev-loop`) で自動実行
- `/loop 20m /addf-dev-loop` で 20分サイクルの自律実行が可能
- 1セッションで 5-6 タスクを完了できる実績あり

## 注意点・制約

- コンパクション（コンテキスト圧縮）前に knowhow を記録しないと知見が消失する
- 大きなタスク（Plan 0005 等）は 20 分サイクルに収まらない場合がある
- Stage 2 のバックグラウンドエージェントは完了を待つ必要があり、次のサイクルに影響することがある
